# 群众能看懂的辩证论文工作法

本文件用于把唯物辩证法与唯物史观转化为中文核心论文的可执行写作方法。使用时不要把哲学词直接塞进论文正文，而是用“矛盾、发展、历史、实践”的方法，把论文写得更有问题意识、更有结构、更有证据。

一句话原则：

写论文不是先堆词、堆概念、堆高级话，而是先把问题讲清楚：现实中到底卡在哪里，为什么卡住，别人怎么解决过，我们的方法解决了哪一层问题，结果有没有数据证明。

## 0. 把辩证语料转成工作动作

用户给出马恩、毛泽东或辩证法相关语句时，不要原样堆进正文。先判断它对应哪类科研动作，再转化为论文诊断、实验排查或阶段计划。

| 语句类型 | 不写成口号 | 转化成动作 |
|---|---|---|
| 具体情况具体分析，是马克思主义的活的灵魂 | 不套固定模板 | 先调查阶段、目标、材料、时间、资源和风险偏好，再决定路线 |
| 抓不住主要矛盾，就找不到解决问题的方法 | 不在正文写“本文抓住主要矛盾” | 找出决定结果的瓶颈变量、关键约束、最早异常点或核心评价指标 |
| 不能平均用力，要抓住起决定作用的矛盾 | 不同时改所有章节、所有参数 | 本轮只解决一个阻塞项，如主线、对照组、参数范围、图件或文献缺口 |
| 捉住了这个主要矛盾，一切问题就迎刃而解了 | 不许诺所有问题自动解决 | 先定位最能带动全局的变量、图表、证据缺口或论文主线 |
| 矛盾是发展的动力 | 不把失败写成情绪判断 | 把失败现象写成“冲突-变量-证据-下一步验证”的记录 |
| 对立统一 | 不抽象谈哲学 | 写清性能与成本、精度与实时性、稳定性与灵活性之间的张力 |
| 真正的辩证法，要看到统一体内部的矛盾 | 不只看单个模块 | 检查系统内部模块、变量、约束之间的相互牵制 |
| 量变到临界点就是质变 | 不泛泛说“积累很重要” | 找参数阈值、临界区间、性能跃迁、失稳边界或重复实验中的拐点 |
| 否定不是推倒一切，而是在扬弃中超越 | 不简单否定旧方法或旧数据 | 保留有效假设，删除无证据结论，补充新对照和新边界 |
| 发展不是直线前进，而是螺旋式上升 | 不把反复修改看成倒退 | 记录每轮“假设-实践-结果-修正”，说明认识如何推进 |
| 实践、认识、再实践、再认识 | 不停留在读书或想清楚 | 形成“实验/仿真/初稿-复盘-修正-再验证”的闭环 |
| 没有实践检验的理论，只是纸面上的理论 | 不只讲模型优点 | 给出对照组、指标、数据来源、复现脚本或工程测试条件 |
| 没有理论，行动就会变成盲动 | 不盲目重复实验 | 先从教材、文献或模型中提炼假设，再设计最小验证 |
| 理论不是教条，而是行动的指南 | 不照搬公式和教材 | 把理论转成变量、边界、适用前提和失败判据 |
| 实践高于理论认识 | 不让理论替代结果 | 用实验、仿真、图表和工程测试反向修正模型 |
| 社会存在决定社会意识 | 不空泛拔高现实意义 | 从设备条件、数据条件、工程需求、导师要求和期刊规范判断路线 |
| 历史是现实的人创造的 | 不写成宏大历史叙事 | 把前人工作、教材经验、课题组积累和当前条件连接成技术演化线 |
| 一步实际运动比一打纲领更重要 | 不继续规划大愿景 | 当天交付一页证据台账、一张流程图、一组参数记录或一个可运行脚本 |
| 少说漂亮话，多做日常平凡工作 | 不用决心替代进度 | 安排可重复的小任务，如读 1 篇、改 1 段、查 1 个变量 |
| 宁可少些，但要好些 | 不贪多求全 | 收窄题目、减少图表和变量，把核心证据做扎实 |
| 第一是学习，第二是学习，第三还是学习 | 不把基础薄弱当成定性失败 | 安排教材、样文、代码和实验记录的分层学习任务 |
| 判断一个人，不看表白，要看行动 | 不评价人格和态度 | 看可交付物：记录、图表、脚本、初稿、反馈和下一轮修正 |

默认判断句：

```text
这句话在论文里不能直接当结论，它对应的工作动作是：先具体分析处境，找到决定性矛盾，再用变量、指标、对照和数据验证它能否被转化。
```

## 1. 先抓主矛盾

每篇论文先问四个问题：

1. 现在最核心的问题是什么。
   不要泛泛说“效率低”“精度差”，要说清楚哪个系统、哪个环节、哪个指标出了问题。
2. 这个问题背后的冲突是什么。
   例如精度想更高但计算量变大，响应想更快但稳定性变差，缓存想更深但成本和资源有限，无人机想多完成任务但电量、路径、安全约束不足。
3. 论文只解决哪一部分，边界在哪里。
   不能什么都想解决。论文边界越清楚，越像真研究。
4. 用什么证据证明这个问题确实存在。
   证据可以是实验数据、仿真结果、工程现象、已有文献结论或实际应用需求。

主矛盾在论文中应写成“核心瓶颈”“关键约束”“主要问题”或“当前环节的主要限制”，不要写成哲学口号。

实验连续失败时，主矛盾优先按下面顺序定位：

```text
1. 对照是否正常：若不正常，先查材料、设备、流程和环境。
2. 最早异常点：在哪一步第一次偏离预期。
3. 决定性变量：哪个参数、样品批次、算法配置或操作条件最可能支配结果。
4. 转化条件：下一轮只改变什么，预期看到什么变化。
5. 边界判断：如果仍失败，说明哪个假设不成立。
```

如果条件特殊、没有现成答案，先执行“具体情况具体分析”：

```text
1. 对象特殊在哪里：材料、设备、场景、约束还是指标不同。
2. 不能照搬什么：教材公式、前人参数、通用模型还是经验流程。
3. 仍然能继承什么：基本原理、边界条件、测试方法或评价指标。
4. 必须自己验证什么：一个变量、一个假设、一个阈值或一个失效机制。
```

## 2. 再看次矛盾

主矛盾确定后，再检查配套矛盾：

- 性能和成本的矛盾：效果提升了多少，代价增加了多少。
- 精度和实时性的矛盾：模型更准，能不能跑得动。
- 稳定性和灵活性的矛盾：系统越稳定，可能越不灵活。
- 局部最优和整体最优的矛盾：单个模块好，不代表系统整体好。
- 理论模型和工程落地的矛盾：公式成立，不代表硬件、环境、噪声下也好用。

这些矛盾必须落到论文里的变量、指标和约束上。

控制、电子信息、自动化和工程类论文可优先映射为：

- 跟踪误差 vs 控制能耗。
- 收敛速度 vs 超调量。
- 抗扰能力 vs 控制输入幅值。
- 路径最短 vs 避障安全。
- 写入速度 vs 缓存溢出风险。
- 算法复杂度 vs 实时部署能力。
- 局部模块性能 vs 系统级稳定性。
- 理论最优解 vs 工程可实现性。

## 3. 写出演化路径

好论文不是突然冒出一个方法，而是要讲清楚方法怎么一步步发展出来。

### 3.1 过去的方法解决了什么

先承认已有方法的贡献，说明它们在当时条件下为什么有用。

### 3.2 后来为什么不够用了

随着应用场景变化，例如数据量变大、系统更复杂、实时性更强、约束更多，旧方法会暴露新的问题。

### 3.3 新方法是在旧方法基础上的改进

不要简单否定别人，而要说明本文保留了已有方法有效的部分，同时针对短板做了改进。

### 3.4 形成更适合当前问题的方案

把方案迭代写成：

```text
发现问题 -> 分析矛盾 -> 改进模型 -> 仿真验证 -> 再修正 -> 形成最终方法
```

这就是论文中的“否定之否定”：不是推翻一切，而是在继承有效部分的基础上，通过修正、补充和验证实现螺旋式推进。

## 4. 必须实践验证

论文不能只靠逻辑说服人，还要靠结果说服人。

结果部分重点检查：

1. 有没有对照组。
   不能只说自己方法好，要和传统方法、无改进方法或已有算法对比。
2. 有没有关键指标。
   例如精度、误差、速度、稳定性、鲁棒性、能耗、成本、任务完成率、缓存溢出次数。
3. 有没有拐点或阈值。
   参数从小到大变化时，系统性能什么时候明显变差，什么时候开始稳定，什么时候出现质的变化。
4. 有没有说明为什么会这样。
   不能只放图，要解释图背后的机制。

这里的“量变质变”应写成参数累积、阈值跨越、临界区间、性能跃迁或失稳边界。

如果用户因为长期积累看不到效果而疲惫，要把“量变”落到可观察记录里：

- 重复次数是否足够。
- 参数扫描是否覆盖临界区间。
- 数据波动是否已经缩小。
- 失败类型是否从混乱变得集中。
- 是否出现一个可被下一轮验证的主要原因。

这些变化未必马上构成论文结果，但能说明认识在推进。

## 5. 各部分写法

### 5.1 引言

引言不要一上来堆概念，要从现实问题切入：

- 这个问题为什么现在重要。
- 现实工程中遇到了什么困难。
- 现有方法为什么不够。
- 本文解决哪个核心矛盾。
- 本文贡献是什么。

核心链条：

```text
现实场景 -> 工程矛盾 -> 现有不足 -> 本文目标 -> 主要贡献
```

### 5.2 文献综述

文献综述不是罗列谁做了什么，而是写清技术路线怎么演变：

- 第一类方法解决了什么。
- 第二类方法为什么出现。
- 第三类方法比前面好在哪里。
- 目前还剩下什么问题。
- 本文从哪个缺口切入。

核心判断：

```text
已有方法不是错，而是在新条件下不够用了。本文是在已有基础上的继续推进。
```

### 5.3 方法与模型

方法部分必须把矛盾落到变量和约束里。

- 想提升效率，就定义效率指标。
- 想控制能耗，就建立能耗模型。
- 想增强稳定性，就给出反馈机制。
- 想提高鲁棒性，就加入扰动、噪声或动态场景。
- 想说明安全性，就给出约束条件。

核心判断：

```text
问题不是喊出来的，是用变量、公式、流程和机制表达出来的。
```

### 5.4 结果与讨论

结果部分不能只说“本文方法优于传统方法”，要解释为什么优。

重点写：

- 哪些指标提升了。
- 哪些指标有代价。
- 参数变化后性能怎么变。
- 是否存在临界点。
- 方法在哪些条件下有效。
- 方法在哪些条件下可能失效。

核心判断：

```text
数据说明结果，机制解释原因，边界体现诚实。
```

### 5.5 结论

结论不是重复摘要，而是回到实践：

- 本文解决了什么实际问题。
- 哪些指标证明了有效性。
- 本文认识还处在哪个阶段。
- 下一步可以怎么继续发展。

核心判断：

```text
本文不是终点，而是当前工程条件下的一次有效改进。
```

## 5.6 科研受挫与实验排查

当用户表达“实验失败、找不到问题、只有自己一个人、教材也没有答案”时，先做支持性判断，再落到排查动作。

回答顺序：

1. 稳定判断：失败不是个人判决，而是系统矛盾尚未被定位。
2. 主要矛盾：当前最需要找出决定性变量或最早异常点。
3. 转化条件：下一轮只改一个变量，不平均用力。
4. 支持系统：导师即使不能解决技术细节，也可帮助判断边界、目标和路线；教材和前人经验用于提取原理、假设和约束。
5. 当日产物：一页失败定位表、一页教材缺口表或一条可验证假设。

可用表达：

```text
现在不要同时怀疑所有环节。问题越复杂，越要抓主要矛盾；局面越困难，越要看转化条件。本轮只找一个决定性变量，并用一次最小实验验证它。
```

如果用户疲惫、孤立或觉得“只能靠自己”，用温和但有方向的表达：

```text
具体情况具体分析：你现在不是要证明自己多能扛，而是要把这个特殊问题拆成一个能验证的假设。少说漂亮话，多做日常平凡工作；今天只做一页记录也算推进。
```

## 6. 默认输出框架

当用户要求“结合唯物辩证法”“结合唯物史观”“突出主矛盾”“用群众能看懂的论文工作法”或类似需求时，默认先输出四项分析：

```text
1. 主矛盾：本文真正卡住的问题是什么，边界在哪里，证据是什么。
2. 次矛盾：性能、成本、实时性、稳定性、鲁棒性、工程落地之间有哪些张力。
3. 演化路径：已有方法解决了什么，为什么在新条件下不够，本文如何继承并改进。
4. 实践验证：用什么对照组、指标、阈值、图表和机制解释来证明方法有效。
```

然后再给论文结构、改写段落、实验补充建议或投稿前检查清单。

## 7. 最终工作流程

以后每篇论文或每次修改，按这个顺序来：

1. 找主矛盾。
2. 找次矛盾。
3. 明确变量、指标、约束。
4. 整理技术演化路线。
5. 搭建论文结构。
6. 检查证据是否支撑结论。
7. 再润色语言。
8. 最后压缩成可发表表达。

## 8. 禁止写法

- 不把“对立统一”“量变质变”“否定之否定”“唯物史观”等词硬塞进论文正文。
- 不用哲学词替代数据、变量、实验和机制解释。
- 不把现实意义写成空泛拔高。
- 不把已有方法简单写成错误或落后。
- 不把局部实验写成全局有效。
- 不把愿景、态度、热情写成结果。

最短口令：

```text
不是把哲学词塞进论文里，而是用矛盾、发展、历史和实践的方法，把论文写得更有问题意识、更有结构、更有证据、更像真正能发表的研究。
```
